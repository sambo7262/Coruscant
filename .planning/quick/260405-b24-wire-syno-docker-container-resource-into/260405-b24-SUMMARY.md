# Quick Task 260405-b24: Wire SYNO.Docker.Container.Resource into fetchNasDockerStats

**One-liner:** Replaced broken Container.list approach with confirmed SYNO.Docker.Container.Resource GET name=any, summing cpu+memoryPercent per container; removed phantom network fields from type and AppHeader.

## What Changed

### Task 1: Type and Adapter

**packages/shared/src/types.ts:** `NasDockerStats` now has exactly 2 fields: `cpuPercent` and `ramPercent`. The `networkMbpsUp` and `networkMbpsDown` fields are removed.

**packages/backend/src/adapters/nas.ts:** `fetchNasDockerStats` completely rewritten:
- Single GET to `SYNO.Docker.Container.Resource` with `name=any` (returns all containers)
- Parses `response.data.data.resources[]` — each item has `cpu` (float %) and `memoryPercent` (float %)
- Sums both across all resources; rounds to 1 decimal
- Fallback to `SYNO.ContainerManager.Container.Resource` on failure
- All old Container.list logic, extractContainers, isRunning, numField helpers, and network aggregation removed

**packages/backend/src/__tests__/nas-adapter.test.ts:** Three `fetchNasDockerStats` tests updated:
- Main aggregation test uses new response shape (resources array with cpu/memoryPercent)
- "both fail" test verifies undefined return after 2 failed attempts
- New fallback test verifies ContainerManager namespace is tried after Docker namespace fails

### Task 2: AppHeader

Removed the `↑{nas.docker.networkMbpsUp.toFixed(1)} ↓{nas.docker.networkMbpsDown.toFixed(1)}` span from the Docker section. The Docker section in AppHeader now shows only CPU % and RAM % lines, plus the image update LED.

Note: `nas.networkMbpsUp` / `nas.networkMbpsDown` on `NasStatus` (the NAS hardware network) remain — these are correctly shown in the middle panel NAS stats section and are unrelated to Docker stats.
