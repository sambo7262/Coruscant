## Copywriting Contract

### Log Viewer — Webhook Category Filter

| Element | Copy |
|---------|------|
| Webhook filter option label | `webhook` (lowercase, consistent with other service filter labels) |
| Webhook log entry format (D-16) | `{service} \| {eventType} \| {title} \| {HH:MM MM/DD/YYYY}` — e.g. `radarr \| Grab \| Movie Title \| 14:30 04/06/2026` |
| Tautulli webhook log entry format | `tautulli \| {event} \| {title} \| {HH:MM MM/DD/YYYY}` |

### Error Log Messages (D-17)

Error messages in the pino logger must be human-readable at a glance. Required format: `[{SERVICE}] {problem description} — {actionable hint}`.

| Context | Current (before fix) | Required (after fix) |
|---------|---------------------|---------------------|
| Adapter poll failure | Raw stack trace or terse code path | `[RADARR] Poll failed: ECONNREFUSED — check base URL in Settings` |
| Auth failure | `401` or raw HTTP code | `[PIHOLE] Auth failed: 401 — re-enter Pi-hole password in Settings` |
| NAS session expired | `session expired` | `[NAS] DSM session expired — adapter will re-authenticate on next poll` |
| Webhook parse error | `JSON parse error` | `[WEBHOOK] Malformed payload from {service} — check webhook configuration` |

**Note:** Exact error message strings are at Claude's discretion per CONTEXT.md "Claude's Discretion" section. The above format pattern is the contract; individual messages should follow it.

## Interaction Contracts

### Log Filter Dropdown — Webhook Category Addition (D-15)

- The category filter `<select>` in `LogsPage` is dynamically derived from loaded entries: `Array.from(new Set(entries.map((e) => e.service)))`
- Adding `service: 'webhook'` to pino log calls in `arr-webhooks.ts` and `tautulli-webhook.ts` automatically populates the filter option — no frontend code change required
- The `'webhook'` option will appear in the dropdown in sort order with other service names (dynamic derivation handles ordering)
- Visual treatment: same `<option>` element style as existing service filter options

### MediaStackRow LED Glow Fix (D-12)

- Glow must be applied via `box-shadow` on the LED element — not `filter: drop-shadow()` (which can cause text bleed)
- Glow color must match the LED color at the time of render (green for idle, amber for warning, red for offline, purple for downloading/queued)
- Animation keyframe for healthy LED glow: `ledBreathe` (3s) — pre-existing in `globals.css`
- The glow intensity must be visually consistent with other tile LEDs at the same health state when viewed from kiosk distance (approximately 2 feet)

## Phase 10 Design Constraints Summary

Constraints carried forward from prior phases that remain binding:

1. **800x480 no-scroll constraint** — dashboard must remain scroll-free at kiosk viewport (Phase 5)
2. **GPU-composited animations only** — all animations use `transform` and `opacity` only, no layout-triggering properties (DASH-08)
3. **JetBrains Mono everywhere** — no serif or sans-serif type introduced; monospace is the aesthetic requirement
4. **CRT scanline overlay** — `body::after` pseudo-element at `z-index: 9999`; no new content may sit above it unless `z-index: 10000+`
5. **Reduced motion respect** — `@media (prefers-reduced-motion: reduce)` in `globals.css` collapses all animation durations to 0.001ms
6. **`--space-deep: #000D1A` as base** — Variant C: Tactical Dark is confirmed and locked (Phase 8 user selection)
