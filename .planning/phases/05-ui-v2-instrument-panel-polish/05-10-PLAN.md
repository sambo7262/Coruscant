<objective>
Extend the arr adapter to poll the queue endpoint and emit download activity metrics.

Current state: `pollArr` only calls `/api/v3/health` and emits `healthWarnings` in metrics. The frontend `MediaStackRow` LED logic reads `metrics.downloading` and `metrics.queue` to decide LED color — but these are never populated by the real adapter (only in mock data). So arr LEDs are always green regardless of download activity.

Fix: Add a parallel call to `/api/v3/queue` in `pollArr`. Extract queue count and active download status. Include in metrics alongside existing `healthWarnings`.

The Radarr/Sonarr/Lidarr/Prowlarr/Readarr queue endpoint is: `GET /api/v3/queue?page=1&pageSize=1&includeUnknownMovieItems=true`
Response: `{ "totalRecords": N, "records": [{ "status": "downloading"|"queued"|"completed"|..., ... }] }`

Output: Arr services with live queue metrics that drive LED color changes in the frontend.
</objective>

### Task 2 — Add queue polling to pollArr

Add a parallel call to `/api/v3/queue` alongside the existing health check. Use `Promise.allSettled` or `Promise.all` with a `.catch(() => null)` fallback so queue failure doesn't break the health check.

Queue endpoint: `GET /api/v3/queue?page=1&pageSize=1`
- `totalRecords` = total queue count
- `records[*].status` — check for any item with `status === 'downloading'`

Extract:
```ts
const queueCount: number   // totalRecords, or 0 on failure
const downloading: boolean // true if any record.status === 'downloading'
const activeDownloads: number // count of records with status === 'downloading'
```

Note: `pageSize=1` only fetches one record — sufficient to detect `downloading` status and get `totalRecords`. If you need accurate `activeDownloads` count, use a larger pageSize or accept that `activeDownloads` may be 0 or 1 when queue is large.

Add these to the metrics object alongside `healthWarnings`:
```ts
metrics: {
  healthWarnings,
  queue: queueCount,
  downloading,
  activeDownloads,
}
```

### Task 3 — Add tests

In `packages/backend/src/__tests__/arr-adapter.test.ts`, add at least 3 tests:

1. **Queue metrics populated:** Mock `/api/v3/queue` returning `{ totalRecords: 5, records: [{ status: 'downloading' }] }` → verify `metrics.queue === 5`, `metrics.downloading === true`, `metrics.activeDownloads === 1`

2. **Queue idle:** Mock queue returning `{ totalRecords: 3, records: [{ status: 'queued' }] }` → verify `metrics.queue === 3`, `metrics.downloading === false`

3. **Queue failure graceful:** Mock queue endpoint returning 500/error → verify service still reports `online` with `metrics.queue === 0`, `metrics.downloading === false` (health check unaffected)
